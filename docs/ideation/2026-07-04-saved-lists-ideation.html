Saved Lists — Ideation

A local-first, personal way to save catalog problems into named lists — collaboration comes later · 4 Jul 2026

Subject: a new feat/saved-lists feature, off main Trigger: the collaborative-first Lists work (PR #10) felt heavy and wrong to use Method: grilling session (grill-me) — resolving the design tree one branch at a time Grounding: in-session

The one thing to take away

PR #10 built lists collaborative-first — every list a cloud, membership-scoped, shareable object, with "personal" as a degenerate one-member case. That put the hardest parts (auth gate, membership, group status, RLS) in front of the simplest, most-used act: save a problem to a list. The reframe: ship a small, local-first, personal "Saved Lists" feature off main — decoupled from #10 — and add collaboration later as a cloud-sync layer, exactly how the app's logbook is local-first with a sync spine bolted on.

This isn't a "redesign" of #10 — it's a new feature on a clean base. The app has no Lists tab on main at all; Saved Lists adds one. #10's collaborative machinery is set aside (to be superseded or folded into the later collaboration layer), not built upon.

Now — Saved Lists

Local SwiftData lists: create / rename / delete, add & remove problems, and a Favorites view. No account, no network, works offline — the thing you'd use solo, every session.

Later — Collaboration

A cloud-sync layer on top (the logbook's pattern): sharing, member roster, one-way promotion, group status. Reuses or adapts #10's Supabase schema. Its own future plan.

Why local-first, off main

The pain — a collaborative-first, cloud-only model taxes the 95% (solo curation) to serve the 5% (a co-located group). You shouldn't need to sign in to jot down a project list.
The inversion — treat the personal list as the primary object and collaboration as an enhancement of it. A list becomes shareable when you choose; until then it's just yours, on your device.
The analogy — the logbook and favorites are already local-first (SwiftData); the logbook grew a cloud-sync spine later (plan 001). Saved Lists follows the same shape: nail the single-user object, then make it multiplayer.
Storage — the one open call. Local-first is recommended and assumed here. Alternatives if cloud-now is preferred: extract #10's backend onto this branch, or rebuild cloud fresh. Only the storage layer changes; the UX is the same.

Carried forward for the later collaboration layer

Decided in the earlier grill (when the feature was collaborative-first); still true, deferred to the cloud phase:

Personal vs shared is explicit — a list becomes collaborative by an explicit action, not by who happens to be in it.
Sharing is opt-in and one-way — a private list stays private until you Share it, which promotes it (no silent demotion).
Member presence on cards — shared lists show member avatars + count.
Open (later): demotion of a shared list back to private — currently one-way. Revisit in the collaboration plan.
